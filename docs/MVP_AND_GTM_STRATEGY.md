# MVP와 초기 사용자 확보 전략

이 문서는 가고시마 여행 앱을 포트폴리오와 상용 서비스 후보로 키울 때, 무엇을 먼저 만들고 어떻게 사용자 반응을 받을지 정리합니다.

핵심 결론:

- 처음부터 많은 기능을 만들지 않는다.
- 사람들이 정말 필요로 하는 한 가지 흐름을 먼저 완성한다.
- 만들기 전부터 누구에게 어떻게 보여줄지 정한다.
- 포트폴리오에서는 기능 수보다 문제 정의, 검증 과정, 배포와 피드백 흐름을 강조한다.

## 1. 이 앱의 진짜 MVP

이 앱의 MVP는 "여행 앱 전체"가 아니다.

가장 작은 핵심은 아래 한 문장이다.

```text
여행 준비자가 여행 정보를 입력하면 부모님이 로그인 없이 공유 링크로 오늘 일정과 다음 이동을 볼 수 있다.
```

따라서 MVP 핵심 기능은 아래 3개만 잡는다.

1. 여행 정보 입력
2. 공유 링크 생성
3. 부모님용 오늘 일정 화면

일정 추천, 앱 내 지도, 실시간 업데이트, 소셜 로그인, 결제, 팀 권한은 MVP가 아니다.

## 2. 만들지 않을 것

초기에는 아래 기능을 의도적으로 만들지 않는다.

- 소셜 로그인
- 결제
- 여러 명이 동시에 편집하는 협업 기능
- 앱 내 길찾기 엔진
- 실시간 위치 공유
- 푸시 알림
- 자동 여행 추천
- 복잡한 관리자 대시보드
- 여러 여행 템플릿

이 기능들은 제품이 커 보이게 만들지만, 사용자가 처음 써야 하는 이유를 증명하지는 못한다.

## 3. Phase별 MVP

### Phase 1. 개인 사례 MVP

목표:
- 부모님 가고시마 여행 사례를 실제로 완성한다.
- 포트폴리오에서 "내가 직접 문제를 발견하고 해결했다"를 보여준다.

필수:
- 부모님용 PWA
- 오늘 일정
- 전체 일정
- 지도 열기
- 체크리스트
- 긴급 정보
- 도메인 배포
- 오프라인 확인

검증:
- 부모님 또는 가족이 앱을 보고 다음 행동을 바로 이해하는지 확인한다.
- 갤럭시 홈 화면에 추가해 실제 앱처럼 실행되는지 확인한다.
- 비행기 모드에서 홈, 일정, 긴급 화면이 보이는지 확인한다.

### Phase 2. 사용자 입력형 MVP

목표:
- 사용자가 직접 여행 정보를 입력하고 공유 링크를 만들 수 있다.

필수:
- 이메일/비밀번호 가입과 로그인
- 여행 생성
- 일정 입력
- 숙소/장소 입력
- 공유 링크 생성
- 공유 링크로 부모님용 화면 조회

검증:
- 다른 사람이 자신의 여행을 10분 안에 만들 수 있는지 확인한다.
- 공유 링크를 받은 사람이 로그인 없이 앱을 이해하는지 확인한다.
- 입력자가 가장 많이 헷갈리는 필드를 기록한다.

### Phase 3. 상용 가능성 검증

목표:
- 가족 여행, 부모님 여행, 골프 여행처럼 반복 가능한 사용 사례가 있는지 확인한다.

검토:
- 여행 템플릿
- 소셜 로그인
- 가족별 공유 권한
- 민감정보 숨김 옵션
- 유료화 가능성
- 광고/홍보 채널

## 4. 초기 사용자 후보

이 앱은 처음부터 불특정 다수를 노리지 않는다.

가장 좋은 초기 사용자는 아래 순서다.

1. 부모님 여행을 준비하는 자녀
2. 골프 여행을 준비하는 가족
3. 효도 여행, 환갑/칠순 여행 준비자
4. 소규모 가족 여행 총무
5. 단체 여행에서 일정을 정리해야 하는 사람

이 사람들은 "내가 여행을 가는 사람"이 아니라 "누군가를 위해 여행 정보를 정리해주는 사람"이다.

## 5. 초기 홍보 전략

영상에서 말한 것처럼 앱은 만들어도 저절로 알려지지 않는다. 이 프로젝트도 처음부터 사용자를 만날 경로를 정해야 한다.

### 포트폴리오 채널

- GitHub README
- 개인 블로그 글
- LinkedIn 또는 커리어 프로필
- 프로젝트 회고 글
- 배포 URL과 데모 계정

강조할 내용:
- 부모님 사용성을 고려한 UX
- PWA와 오프라인 대응
- Go 백엔드와 PostgreSQL 확장 설계
- 로그인 없는 공유 링크
- 민감정보 공개 범위 설계

### 실제 사용자 검증 채널

- 가족/친척 여행 준비자에게 직접 보여주기
- 골프 모임이나 부모님 여행을 준비하는 지인에게 데모 링크 보내기
- 커뮤니티에 "부모님 여행 일정 공유 도구"로 소개하기
- 실제 여행 사례를 익명화해서 블로그에 기록하기

초기 메시지 예시:

```text
부모님 여행 일정, 숙소, 지도 링크, 긴급 연락처를 한 화면에서 볼 수 있게 정리하는 PWA를 만들고 있어요.
여행 준비하는 사람이 입력하면 부모님은 로그인 없이 링크만 열면 됩니다.
```

## 6. 피드백 질문

사용자에게 기능 아이디어를 먼저 묻기보다, 실제 불편을 묻는다.

좋은 질문:

- 부모님 여행 정보는 보통 어디에 정리하시나요?
- 부모님께 숙소 주소나 지도 링크를 어떻게 전달하시나요?
- 여행 중 부모님이 가장 자주 물어보는 정보는 무엇인가요?
- 공유 링크를 받으면 로그인 없이 보는 방식이 편한가요?
- 직접 여행 정보를 입력한다면 가장 귀찮아 보이는 부분은 무엇인가요?

피해야 할 질문:

- 어떤 기능이 있으면 좋을까요?
- 소셜 로그인 필요할까요?
- 앱 안에 지도가 있으면 좋을까요?

기능 질문은 답이 너무 넓어진다. 대신 실제 행동과 불편을 묻는다.

## 7. 성공 기준

### 개인 사례 성공

- 부모님 폰에서 홈 화면 앱처럼 실행된다.
- 다음 일정과 지도 버튼을 바로 찾는다.
- 오프라인에서도 핵심 정보가 보인다.
- 가족에게 보여줬을 때 "이거 우리 여행에도 쓰고 싶다"는 반응이 나온다.

### 사용자 입력형 MVP 성공

- 사용자가 새 여행을 직접 만든다.
- 일정과 숙소를 최소 1개 이상 입력한다.
- 공유 링크를 다른 사람에게 보낸다.
- 공유 링크를 받은 사람이 로그인 없이 정보를 확인한다.
- 입력자가 다시 수정하러 돌아온다.

### 상용화 후보 성공

- 같은 문제를 가진 사용자가 반복적으로 발견된다.
- 여행 준비자가 돈을 내거나 시간을 아끼는 가치가 명확하다.
- 단순 여행 일정표보다 "부모님/가족 공유"라는 차별점이 이해된다.

## 8. 다음 작업 우선순위

1. 부모님용 개인 사례를 완성한다.
2. 포트폴리오 README와 회고 글을 준비한다.
3. 사용자 입력형 MVP를 위한 인증/여행 생성 API를 만든다.
4. 관리자 입력 화면을 아주 작게 만든다.
5. 공유 링크 조회 화면을 만든다.
6. 가족/지인 3명에게 실제로 써보게 한다.
7. 피드백을 보고 기능을 늘릴지, 입력 흐름을 줄일지 결정한다.

## 9. 포트폴리오에서의 표현

이 프로젝트는 아래 문장으로 설명할 수 있어야 한다.

```text
부모님 여행을 준비하는 사람이 일정과 장소를 입력하면,
부모님은 로그인 없이 공유 링크로 오늘 일정, 지도, 긴급 정보를 확인할 수 있는 모바일 PWA입니다.
```

기술보다 먼저 문제를 설명하고, 그 다음 기술 선택을 설명한다.

```text
왜 PWA인가?
  앱스토어 없이 부모님 폰에 빠르게 설치해야 했기 때문입니다.

왜 공유 링크인가?
  부모님에게 로그인 과정을 요구하면 사용성이 떨어지기 때문입니다.

왜 Go 백엔드인가?
  사용자별 여행 데이터, 인증, 공유 링크, API 서버를 단순하고 명확하게 구현하기 위해서입니다.
```
